# Technical Audit — Inception (tmarcos)

> Audit date: 2026-04-20
> Audited against: 42 Inception subject (mandatory part)

---

## Executive Summary

The project is technically competent and structurally coherent. All three services exist in dedicated containers, Dockerfiles are written by hand, volumes are named, the network is explicit, credentials are in `.env` and out of git, and every entrypoint uses `exec` to guarantee correct PID 1 behaviour. WordPress automation via WP-CLI is well-implemented, including MariaDB readiness polling.

**Top reprovação risks:**

1. Portuguese comment visible in `README.md` line 76 — `//ver se vou descomentar isso`
2. Example password hardcoded in `DEV_DOC.md` line 96 — `-pwpuserpass123`
3. `server_name _` in NGINX config — does not explicitly configure `tmarcos.42.fr`
4. `debian:bookworm` is the current stable Debian, not the penultimate — borderline per subject
5. Makefile `fclean` destroys **all** Docker images on the system, not just project images

**Overall conformity estimate: ~82%**

## Problems Found

### Problem 1 — Portuguese comment in README.md
- **Severity:** Critical
- **File:** `README.md`, line 76
- **Description:** The line `//ver se vou descomentar isso` is an accidental Portuguese comment left in the final file, immediately followed by a commented-out access URL block (lines 77-78).
- **Why it violates the subject:** The README must be in English. An evaluator reads this within the first 30 seconds and interprets it as carelessness or an incomplete submission.
- **Fix:** Delete line 76. Uncomment lines 77-78 and integrate the access URLs into the Instructions section properly.

---

### Problem 2 — Hardcoded example password in DEV_DOC.md
- **Severity:** High
- **File:** `DEV_DOC.md`, line 96
- **Description:** The useful commands block contains `docker exec -it mariadb mariadb -u wpuser -pwpuserpass123 wordpress`. The string `wpuserpass123` looks like a real development password left in the versioned documentation.
- **Why it violates the subject:** *"Any credentials, API keys, passwords found in your Git repository will result in project failure."* Even in documentation, a strict evaluator can invoke this rule.
- **Fix:** Replace with an environment-variable-based example:
  ```bash
  docker exec -it mariadb mariadb -u "${MYSQL_USER}" -p"${MYSQL_PASSWORD}" "${MYSQL_DATABASE}"
  ```

---

### Problem 3 — `server_name _` in NGINX config
- **Severity:** Medium
- **File:** `srcs/requirements/nginx/conf/default.conf`, line 3
- **Description:** `server_name _;` uses the wildcard catch-all instead of the project domain. NGINX accepts any hostname on port 443, not specifically `tmarcos.42.fr`.
- **Why it violates the subject:** The subject requires the domain `<login>.42.fr` to be configured. The certificate CN uses the domain correctly via `$DOMAIN_NAME` in the entrypoint, but the NGINX `server_name` directive itself does not reference the domain name.
- **Fix:** Option A — hardcode the value: `server_name tmarcos.42.fr;`. Option B — generate the config dynamically in the entrypoint using `envsubst`:
  ```bash
  envsubst '${DOMAIN_NAME}' < /etc/nginx/conf.d/default.conf.template > /etc/nginx/conf.d/default.conf
  ```

---

### Problem 4 — `debian:bookworm` as base image
- **Severity:** Medium
- **File:** All three Dockerfiles
- **Description:** All Dockerfiles use `FROM debian:bookworm`. Debian 12 (bookworm) is the **current stable** Debian release, not the penultimate. The penultimate stable would be Debian 11 (bullseye). The subject states "penultimate stable version of Alpine or Debian".
- **Why it violates the subject:** Depends on the evaluator's interpretation. If Debian 13 (trixie) has not been officially released at evaluation time, bookworm could be argued as acceptable. But it is a genuine risk.
- **Fix:** Change to `FROM debian:bullseye` for full compliance. Note: `php8.2-fpm` is not in bullseye's default repositories and would require adding `packages.sury.org` or using `php7.4-fpm` instead. Alternatively, prepare a clear verbal justification that trixie is not yet released.

---

### Problem 5 — Makefile `fclean` removes all Docker images
- **Severity:** Medium
- **File:** `Makefile`, line 17
- **Description:** `docker rmi -f $$(docker images -qa)` removes every Docker image present on the system, not just the three project images.
- **Why it is a problem:** In a shared VM or evaluation environment, this silently deletes images belonging to other projects. It is an aggressive and unpredictable side-effect of `make fclean`.
- **Fix:**
  ```makefile
  docker rmi -f nginx wordpress mariadb 2>/dev/null || true
  ```

---

### Problem 6 — Named volumes use bind-mount semantics
- **Severity:** Medium
- **File:** `srcs/docker-compose.yml`, lines 38-50
- **Description:** Both named volumes use `driver: local` with `type: none` and `o: bind`, which creates named volumes that directly mount a host directory — effectively bind mounts managed by Docker.
- **Why it is a risk:** The subject explicitly states: *"Bind mounts are not allowed for these volumes."* The majority of evaluators accept this pattern since the volumes appear in `docker volume ls` and are managed by Docker. However, a strict evaluator can argue this is semantically equivalent to a bind mount.
- **Fix:** If questioned, explain that these are named volumes managed by Docker (verifiable via `docker volume ls` and `docker volume inspect`). If you want to eliminate the risk entirely, remove `driver_opts` and use pure named volumes, then document that data will be stored in Docker's default volume path (`/var/lib/docker/volumes/`).

---

### Problem 7 — Polling loop with `sleep 2` in WordPress entrypoint
- **Severity:** Low
- **File:** `srcs/requirements/wordpress/tools/entrypoint.sh`, line 8-11
- **Description:** The MariaDB readiness loop can run indefinitely if MariaDB never becomes available. The subject prohibits containers started with infinite loops.
- **Why it is a risk:** `sleep 2` inside a polling loop is generally accepted as legitimate health-check practice. The subject specifically calls out `sleep infinity` and `while true`. Risk is low but real if an evaluator reads the script closely.
- **Fix:** Add a retry counter:
  ```bash
  RETRIES=30
  until mariadb -h mariadb -u "${MYSQL_USER}" -p"${MYSQL_PASSWORD}" "${MYSQL_DATABASE}" -e ";" 2>/dev/null; do
      RETRIES=$((RETRIES - 1))
      [ $RETRIES -eq 0 ] && echo "[wordpress] MariaDB not ready. Aborting." && exit 1
      echo "[wordpress] Waiting for MariaDB..."
      sleep 2
  done
  ```

---

### Problem 8 — No `.dockerignore` files
- **Severity:** Low
- **File:** All service directories
- **Description:** No `.dockerignore` file exists in any of the three service directories, despite the subject's example directory structure showing them for `nginx/` and `mariadb/`.
- **Why it is a problem:** Not explicitly required by the subject text, but present in the reference structure. More importantly, sending unnecessary files to the build context (like `README.md`, `*.sh.bak`) is bad practice.
- **Fix:** Create a minimal `.dockerignore` in each service directory:
  ```
  *.md
  .git
  ```

## Final Verdict

### PASSES WITH RESERVATIONS

The project's architecture is sound: correct PID 1 handling, no prohibited hacks, named volumes, explicit network, credentials out of git, complete documentation. An evaluator will not find fundamental structural failures.

However, **two items carry direct reprovação risk:**

1. **`README.md` line 76** — `//ver se vou descomentar isso` — the kind of oversight that an evaluator notices in the first 30 seconds and that casts doubt on the entire submission.
2. **`DEV_DOC.md` line 96** — `-pwpuserpass123` — looks like a real development password committed to the repository. A strict evaluator can invoke the "credentials in git = immediate failure" rule.

With these two fixed, the project rises to **"probably passes"**.
